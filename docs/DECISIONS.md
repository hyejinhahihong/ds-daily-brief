### 물량: 티어별 가변 쿼터
- 8×3=24건 고정(레퍼런스 방식)을 **거부**함.
- 근거: 레퍼런스 뉴스레터 출처를 섹션별로 분석한 결과,
  **거의 모든 섹션이 [원 소스 1건] + [준수한 블로그 1건] + [SEO성 벤더 블로그 1건]** 구조였다.
  "3건 고정"이 3번째 슬롯을 쓰레기로 채우게 만든다.
- → 가변 쿼터 채택. **빈 섹션은 결함이 아니라 "그 분야가 조용하다"는 신호.**
- Phase 1 stub 검증에서 FDE 0건이 실제로 비워짐 = 이 원칙이 코드에서 지켜짐 확인.

### 소스 판정: "벤더냐"가 아니라 "원 소스냐"
- 처음엔 벤더 블로그를 배제하려 했으나 **사용자가 반박**했고, 반박이 옳았다.
- `devblogs.microsoft.com`이 자사 프레임워크를 발표 → Tier 1 (벤더지만 원 소스)
- `techsy.io`가 남의 벤치마크를 재편집 → Tier 5 (비벤더지만 2차 가공)
- **Tier 5는 배제가 아니라 페널티(×0.5).** 가변 쿼터가 있으므로 자연 도태된다.

### 2축 분리 (혼동 금지)
- `lane_weight` = 관심도, `source_tier` = 원본성. **독립 축.**
- 예: GitHub Releases는 원 소스라 tier 1이지만 대부분 버그픽스라 weight 0.4.
- `lane_weight`는 Phase 0 실측만으로 튜닝하지 않는다 — 관심도는 랭킹 진입률(선별/수집)을 봐야
  판정 가능하므로 Phase 1 실제 실행 후 사용자와 합의해 조정한다.

### lane_weight 전부 동결 (Phase 1 실행 후 확정, 2026-07-16)
- Phase 1 실제 Haiku 랭킹의 진입률·평균 base_score를 보고 **변경 없이 동결**로 확정.
- 근거: 진입률 분포가 설계 의도와 일치한다.
  - 학회(레인2, w0.85) 진입률 **29%·평균 base 6.9**로 1위 — T2 쿼터를 정당하게 선점. 설계대로.
  - arXiv 일반(레인4, w0.7) 143건 수집·진입 1%는 **결함이 아니라** T2에서 학회에 밀린 정상 동작.
  - 미디어(레인6, w0.55) 진입 0%의 원인은 weight가 아니라 **평균 base 2.1**(Haiku가 2차 가공
    뉴스를 낮게 채점). weight를 만져도 안 풀린다. T1 쿼터는 빅테크가 이미 채우고,
    미디어는 빅테크가 조용한 날의 폴백으로 작동하게 둔다.
- → weight는 관심도 축이고, 이번에 드러난 문제는 전부 base(품질) 축이므로 weight 조정 대상 아님.

### Tier 5 자동 판정 안 함 — 화이트리스트 유지 (Phase 1, 2026-07-16)
- Tier는 **수집 시점 도메인/피드 화이트리스트**로 정적 부여(config/sources.yaml). LLM 판정 아님.
- 자동 판정(LLM 등)을 **도입하지 않기로** 확정. 근거: 레인 7개가 전부 명시적 화이트리스트라
  2차 가공(Tier 5) 소스가 유입될 경로가 애초에 없다. Phase 1 실측에서도 Tier 5 = 0건.
- Tier 5 페널티(×0.5) 로직 자체는 **제거하지 않는다** — 향후 웹서치 레인(§10) 활성화 시
  2차 가공이 유입될 수 있으므로 그때를 위한 보험으로 유지.

**v4 → v5 개정 근거 — `fde` → `practice` 확장·개명 (사용자 확정, 2026-07-16).**

- **rung 확장·개명 (FDE 실무사례 → 실무 사례·노하우).** 구 FDE는 정의(Forward Deployed Engineer)가 좁아
  Phase 1 실측 **0건**이었고, **실무자 회고·워크플로·인터뷰·팁**이 갈 곳이 없어 기본값(예측 모델링)으로
  떨어졌다. → 정의를 **"어떻게 일하나"**로 넓히고(작업 방식·시행착오·회고·팁 + FDE + 현장 구축), **T3→T2 승격**.
  레인 5(실무 블로그) 25건/일이 주 공급원. slug `fde`→`practice`, 경로 `/category/practice.html`.
- **사다리 하단(rung 8) 배치.** 다른 7개는 **주제 축**(무엇에 관한 글), practice만 **형식 축**(어떻게 쓰인 글).
  "Airbnb가 LLM 평가를 하루로 단축한 회고"는 MLOps 주제이면서 노하우 형식 — **형식 축을 위에 두면 실무 글을
  전부 흡수**해 예측 모델링이 자석이던 문제가 재발한다. → practice를 맨 아래로 내려 주제 rung(1~7)에서 갈 곳
  없는 형식-글만 담게 하고, 최종 기본값은 7번(예측 모델링)으로 둔다. 8번 과다는 형식 축 자석화 신호로 경고.
- **레인 상한(진단 4-a/b) 폐기.** 채택 안 함. 근거: 레인 상한은 **자리를 뺏을 뿐 무엇이 들어올지 통제하지
  못한다.** 빅테크 2건을 빼도 base 높은 학회 논문이 그 자리를 먹는다. **자리를 뺏는 것과 자리를 만드는 것은
  다르다** — 카테고리 쿼터가 옳은 도구다(자리를 만든다). 진단도 생략.
- **새 카테고리(9번째) 추가 안 함, 8개 유지.** 근거: ① 8개에서도 구 FDE가 비어 있었다(자리 부족이 문제가
  아니었다). ② **주제 축 카테고리에 "회사/형식 축"을 섞으면 라우팅이 붕괴**한다 — practice를 형식 축으로 두되
  사다리 맨 아래에 격리하는 것으로 충분하다.
- **프런티어 랩 신규 발표는 rung 3에서 정상 처리됨.** 별도 "발표" 카테고리 불필요. 증거: GPT-Red(OpenAI

## 레인 base_score 분포 실측 (2026-07-16) — **측정만, 결정 보류**

레인 5 실무 블로그가 25건 수집→0건 선별이 계속되는 원인 진단. **읽기 전용, 결정 안 함**(며칠 운영
데이터 후 사용자가 정한다). raw 295건 Haiku 랭킹 1회 실측:

```
레인               n    평균  중앙   최고   판정
1. 빅테크리서치     27   4.61  5.00   7.0
5. 실무블로그       25   5.42  5.50   7.5   ← Haiku 는 높게 본다
6. AI미디어         62   2.42  2.25   6.5   ← 정당한 저평가(2차 가공)

content_type       n    평균  중앙
paper             165   6.25  6.30
blog               55   4.87  5.50
news               74   2.86  2.50
```

- **핵심: 레인 5는 base 중앙 5.5 — "정당한 탈락(2점대)"이 아니라 "weight에 의한 진짜 손실"이다.**
  (가설은 "2점대면 Haiku가 옳게 본 것"이었는데, 실측은 5점대 → weight 문제.)
- 구조: 빅테크 `1.0×1.0=1.00` vs 실무블로그 `0.6×0.85(tier3)=0.51`. 실무 블로그는 **빅테크의 2배 base를
  받아야** 이긴다. 실제로 레인 1 중앙(5.00)보다 레인 5 중앙(5.50)이 높은데도 final 에서 밀린다.
- 레인 6(미디어)은 중앙 2.25 → weight 아닌 **품질**이 문제(정당한 저평가). 레인 5와 성격이 다르다.
- content_type: blog 4.87 은 0 바닥이 아니다(재현성·정형데이터 감점이 있어도 blog가 구조적 0점은 아님).
  단 paper(6.25)와 1.4점 격차는 존재.
- **시뮬레이션**: 레인 5 상위 5건을 weight 0.85로 계산하면 가정 final 4.7~5.4 로 **전부 컷오프(4.0) 통과**.
  현재 0.6에선 3.3~3.8 로 탈락. → weight 가 유일한 결정 변수임이 재확인.
- **조치 안 함.** lane_weight/tier 는 며칠 실운영 데이터를 보고 사용자가 결정. 이 표는 그 결정의 입력.

## UTC/KST 날짜 버그 (2026-07-17 실측·수정)

- **증상**: 2026-07-17(금) 06:30 KST 자동 실행이 `/news/2026/07/16.html`에 덮어씀. 라벨이 하루 밀림.
- **원인**: **GitHub Actions 러너 타임존은 UTC.** cron `30 21 * * 0-4` = UTC 목 21:30 = KST 금 06:30인데,
  코드가 naive `date.today()`를 써서 UTC 기준 07-16 을 반환 → 매일 "어제 날짜"로 저장.
- **원칙 (추가): "Actions 러너는 UTC다. 모든 날짜는 KST로 계산한다."** `src/config.py`에 `KST=ZoneInfo("Asia/Seoul")`,
  `now_kst()`/`today_kst_iso()` 헬퍼. 날짜라벨을 만드는 naive `date.today()`를 전부 교체(run_daily·dedup·
  run_phase1/2·run_collect), 절대시각(generated_at·collected_at·수집창)도 KST-aware 로 통일. **외부 피드/기사
  published_at 파싱만 원 소스 tz(UTC 가정) 유지** — 그건 남의 데이터라 우리 tz로 바꾸면 안 된다.
- **이중 안전장치**: 워크플로 잡에 `env: TZ: Asia/Seoul`. 커밋 메시지 날짜도 KST(`date` 직접).
- **함정**: Windows/일부 환경은 시스템 tz DB 가 없어 `ZoneInfo("Asia/Seoul")`가 `ZoneInfoNotFoundError`.
  → `tzdata` 의존성 추가(pyproject). ubuntu 러너는 시스템 tz 있어도, 이식성 위해 명시.
- **기존 파일 정리**: git 커밋 시각(UTC)으로 정체 판별 — 4898c84(07-16 11:55 UTC=20:55 KST)=수동 dispatch=진짜
  07-16(16건), cf7edd7(07-16 22:30 UTC=07-17 07:30 KST)=cron=실제 07-17(14건, 버그로 07-16 라벨). 겹침 0.
  → cf7edd7 을 `published/2026-07-17.json`(run_date 교정)으로, 4898c84 를 `published/2026-07-16.json`으로
  복구 후 build_site 재생성. **날조 아님 — git 기록된 실제 데이터의 라벨만 교정.**
